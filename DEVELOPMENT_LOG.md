# Wine/AviUtl2 アダプタ 開発総括

> 全パッチ適用済みの `wine/` ソースツリーと CLAUDE.md の記録から、各課題について「症状→試行錯誤→真因→修正」を抽出した walkthrough。

---

## 課題1: 動画表示の緑がかり

**症状**: AviUtl2 で動画を再生すると、画面全体が緑がかって表示される。

**最初の仮説（誤り）**: CLAUDE.md の古い記述を鵜呑みにし、「NV12 フォーマット（H.264 デコード出力の標準的な YUV フォーマット）の Y 平面/UV 平面マッピングが誤っている」と考えていた。

**なぜその仮説が外れたと分かったか**: `WINEDEBUG=+d3d11,+d3d` でテクスチャ作成ログを取得し、実際に作られているフォーマットを集計した。結果、**NV12 フォーマットのテクスチャは 1 つも作成されていなかった**（A8_UNORM 315回、R8G8B8A8_UNORM 35回、G8R8_G8B8_UNORM 1回など）。仮説の前提そのものが崩れた。

**真因**: `dlls/wined3d/utils.c` の `convert_g8r8_g8b8_unorm` 関数。G8R8_G8B8 フォーマット（パックド YUV 4:2:2、マクロピクセル `[Y0, Cb, Y1, Cr]`）を GL_RG8（2チャンネル）に変換する際、**Cr（赤色差）成分を捨てていた**。シェーダー側は Cr=0 として YUV→RGB 変換を行うため、緑成分(G)が過剰にブーストされる。

**修正**: 変換先を GL_RG8(2ch) から GL_RGBA8(4ch) に変更し、`[R=Cb, G=Y, B=Cr, A=255]` で全チャンネルを保持。OpenGL 側で解決した後、Vulkan バックエンドでは同じ変換ロジックが呼ばれていないことが判明し、`vulkan_formats[]` テーブルと `format->upload` の初期化を別途 Vulkan 側にも追加して両対応させた。

**教訓**: 既存ドキュメントの原因分析を検証せずに次の修正へ進まなかったこと。ログという一次情報に当たって前提を確認したことで、無駄な修正の積み重ねを避けられた。

## 課題2: Vulkan バックエンドの黒画面・UI チラつき

**症状**: Vulkan バックエンド使用時、ウィンドウの一部/全体が黒く点滅する。OpenGL バックエンドでは発生しないが、OpenGL は動作が重すぎて実用に耐えない。

**試行と失敗の系譜**:

| 仮説 | 内容 | 結果 |
|---|---|---|
| ① vsync 設定 | MaxFrameLatency, CSMT 等のレジストリ変更 | 効果なし |
| ② oldSwapchain 管理ミス | スワップチェイン再作成時の引き継ぎ不備 | 部分的に改善したが根本解決せず |
| ③ バックバッファ保持方法 | DISCARD 時に VK_IMAGE_LAYOUT_UNDEFINED を使い前フレーム内容が消える | 発見・修正（有効だった） |
| ④ X11 WSI 層の問題と誤診断 | 「wined3d Vulkan+X11 の WSI 層に原因があり、wined3d 側では直せない」と一時結論 | **誤りと判明**（後述） |

**誤診断からの回復**: ③の UNDEFINED レイアウト問題を修正した後もチラつきが続いたため、「これは Wine 本家の Vulkan WSI 実装の既知の問題で、自分たちの手には負えない」と一度結論づけた。回避策として OpenGL へのフォールバックまで提案していた。

しかし実際にビルドし直して調査を続けたところ、**真因は全く別の場所にあった**: `adapter_vk.c` の `adapter_vk_uninit_3d` 関数で**閉じ括弧 `}` が1つ欠落**しており、不完全な状態でオブジェクトファイルがリンクされ、`vkCreateSwapchainKHR` 自体が失敗していた。さらに調査中に `git checkout` を誤って実行し、パッチ済みの `swapchain.c` を消失させるという別のミスも発生し、複数の問題が同時に絡み合って原因の切り分けを困難にしていた。

**真因（最終）**: ビルドの破損（構文エラーによるリンク不整合）。設計上のロジックエラーではなかった。

**教訓**: 「これは自分の手に負えない外部要因だ」という結論は、実は単純なビルドエラーを隠していた。原因を環境や外部ライブラリのせいにする前に、まず自分のビルド・デプロイパイプラインが正しく機能しているかを疑うべき。

## 課題3: Vulkan バックエンドの「1フレーム遅れ」問題

**症状**: 再生中の映像が1フレーム分古い状態で表示される。マウスホバー時、直前にホバーしていたボタンがハイライトされたままになる。

**試行の系譜**:
1. `invalidate_location(TEXTURE_RGB)` を追加 → 効果不明、後に「載せる場所を間違えていた」バグと判明
2. `vkQueueWaitIdle` を追加して GPU 同期を強制 → 改善の兆候はあったが根本解決せず、パフォーマンス低下
3. GDI パス（システムメモリ経由の描画）に一時的に切り替えて比較 → GDI では正しく表示される。これにより **問題が Vulkan blit 以降の層にある** と切り分けられた
4. `bb_image`（バックバッファの VkImage ハンドル）を毎フレームログ出力 → 値は変化せず一定。ローテーション自体は正しく起きている
5. render pass の subpass 依存関係を修正 → 未解決のまま

**現状**: このタスクは「解決」まで到達しておらず、複数のデバッグ手法（ログ出力によるハンドル追跡、GDI との比較切り分け、GPU 同期の強制）を試した記録が残っている。最終的に `swap_interval==0` 時に `VK_PRESENT_MODE_IMMEDIATE_KHR` を優先する設定変更で改善が見られたが、完全解決とは言い切れない状態。

**教訓**: すべての課題が「解決」で終わるわけではない。原因を1つの層（Vulkan blit）に切り分けられた時点で、それ自体が価値のある成果である。

## 課題4: GCMZDrops（ドラッグ＆ドロップ）関連のクラッシュ

**症状**: ファイルを AviUtl2 にドラッグ＆ドロップすると、`DragEnter フック処理に失敗しました` エラーおよびフリーズ・page fault クラッシュが発生。

**真因1**: `SHCreateDataObject`（shell32.dll）が Wine では `E_NOTIMPL` のスタブのままだった。GCMZDrops プラグインがこの未実装 API を呼んでいた。
→ `IDataObject_Constructor` を使った実装を追加。

**真因2**: `DestroyWindow(hwnd)` をウィンドウプロシージャ内で呼んだ後、同じ `hwnd` に対して `DefWindowProcW` を呼んでしまい、破棄済みの無効ハンドルにアクセスしてクラッシュ（`c0000005`）していた。
→ `DestroyWindow` 呼び出し後は `DefWindowProcW` を呼ばないようガード追加。

**真因3**: `RenderFILENAMEA`/`RenderFILENAMEW` が、pidl 配列が空の場合に `apidl[0]` へ NULL ポインタアクセスしていた。
→ 空配列のケースをハンドリングする NULL ガード追加。

**教訓**: 1つの症状に対して原因が複数（未実装 API、ウィンドウ破棄後アクセス、NULL 配列アクセス）重なっているケース。1つ直しても別の原因でクラッシュが再現し続けたため、都度新しいクラッシュログを取得し、失敗箇所を1つずつ切り分けていく必要があった。

## 課題5: ファイルマネージャ統合（My Computer / Favorites）

**症状**: Wine 標準のシェル名前空間に、Linux のマウントドライブとブックマークを表示・操作可能にしたい。

**この課題は特に「1つ直すと別が壊れる」の連鎖が多かった**:

1. マウントドライブを My Computer 内に表示 → 表示はされたが中身（無名フォルダ、コントロールパネル重複）が壊れていた
   - 原因: PIDL のアロケーションサイズ計算ミス（`sizeof(SHITEMID)` は `abID[0]` を含むため `+1` は不要だったが加えていた）
2. サイズ計算を直したら別の箇所で **二重解放（double free）** が発生
   - 原因: `AddToEnumList` は PIDL の所有権ごと保存する（コピーしない）仕様だったが、呼び出し側でも `ILFree` していた
3. マウントドライブの子フォルダに移動できない
   - 原因: `BindToObject` が PIDL チェーンの最初の要素だけ処理し、後続の子フォルダ要素を無視していた
4. Favorites のブックマーク（.lnk ファイル）をクリックしても開けない
   - 複数のパス表現（`Z:\` 形式、`\\?\unix\` 形式）を試行錯誤し、最終的に `.lnk` の PIDL 格納方法を修正して解決
5. My Computer を開くとフリーズする副作用が発生
   - 原因: `.lnk` 検出ロジックの中で `SHGetIDListFromObject` を My Computer（仮想フォルダ）に対しても呼んでしまい、内部でネットワーク越しの名前解決待ちが発生していた
   - 修正: フォルダ属性（`SFGAO_FOLDER`）を先にチェックし、仮想フォルダの場合は `.lnk` 検出処理自体をスキップ

**さらに後日判明した問題**: ネットワーク共有フォルダ（cifs/nfs/smbfs）がマウントされている環境では、`_access()` によるアクセス確認がネットワーク越しの通信リトライで止まってしまい、シェル全体がフリーズする現象があった。`is_network_fs()` 判定を追加して `_access()` チェックをスキップする形で解決した。

**教訓**: シェル名前空間の実装は、1つの機能追加が既存の別機能（この場合 My Computer の通常動作）を破壊するリスクが高い領域。修正のたびに「新しく直した機能」だけでなく「壊れていないはずの既存機能」も含めて再テストする必要があった。

## 課題6: Space キーで動画再生を停止できない（未解決・調査中）

**症状**: 動画再生中、Space キー（またはマウスクリック）での一時停止が効かない。再生終了後に、押していたはずの操作が一括で実行される。

### 仮説の変遷（すべて「棄却の理由」付き）

| # | 仮説 | 検証方法 | 結果 |
|---|---|---|---|
| 1 | メインスレッドが D3D Present コールでブロック | `vkAcquireNextImageKHR` のタイムアウトを無限→0 に変更 | 効果なし。ブロックが原因ではない |
| 2 | IMMEDIATE present モードでスワップチェインが詰まる | MAILBOX モードを優先に変更 | 効果なし |
| 3 | `IsDialogMessageW` に VK_SPACE ハンドラが未実装 | Wine ソースに実装追加 | AviUtl2 は IsDialogMessageW を呼んでいないため無効 |
| 4 | CS キューのスピンループがメッセージを処理しない | `wined3d_cs_mt_finish` 等に PeekMessage ポンプ追加 | 安定するが効果なし。後にこの経路自体に再生中は到達しないと判明 |
| 5 | Present 直後のメッセージポンプ | `wined3d_swapchain_present` の mutex unlock 直後に DispatchMessageW | **クラッシュ**（再入: WndProc → 停止処理 → D3D Present → wined3d_swapchain_present） |
| 6 | WH_GETMESSAGE フックで割り込み | `SetWindowsHookExW` で PeekMessage に割り込み | **クラッシュ**（page fault。全メッセージに発動しすぎた） |
| 7 | PeekMessageW 内で PostMessageW 再投入 | PM_REMOVE で VK_SPACE 取得時、PostMessageW で再投入し WM_NULL を返す | **無限ループ**（再投入→再取得→再投入）。lParam ガードで解消したが停止不能は変わらず |
| 8 | PeekMessageW 内で DispatchMessageW | VK_SPACE 検出時に直接 dispatch_message を呼ぶ | AviUtl2 の WndProc は WM_KEYDOWN を受け取るが **再生中は無視** |

### トレースによる誤った思い込みの訂正

d3d_perf トレースを取得し、`wined3d_cs_mt_finish` の呼び出し元スレッドを特定したところ、**当初「メインスレッドがここでブロックされている」と考えていたが、実際にブロックされていたのはビデオデコードスレッド（別スレッド）だった**ことが判明。メインスレッドは再生中に wined3d でブロックされていなかった。

### 確定した事実（2026-07-10）

`PeekMessageW` 内に `GetTickCount()` ログを仕込んで再生中に Space を押したテストにより確定:

- **AviUtl2 は再生中に `PeekMessageW` を全く呼ばない**
- メッセージは再生終了後にキューから一括取得される
- `DispatchMessageW` で WndProc に直接 VK_SPACE を送っても、AviUtl2 は再生中はそれを無視する
- user32.dll レベルの介入は再生中に到達しないため根本的に無効

### 残されたアプローチ

wined3d の毎フレーム到達箇所（`wined3d_swapchain_present` 等）からの介入。ただし D3D コールスタック上での `DispatchMessageW` は再入クラッシュが避けられない点が課題。

**現状ステータス**: 未解決。AviUtl2 が再生中に PeekMessageW を呼ばない・WndProc に WM_KEYDOWN を送っても無視するという2つの障壁が確定しており、根本的な解決には AviUtl2 内部の再生ループ構造のさらなる分析が必要。

## 全体を通しての開発プロセス上の教訓

1. **既存ドキュメントの記述を検証してから次に進む**（課題1）: CLAUDE.md に書かれた「原因」が間違っている可能性を常に疑い、ログという一次情報で裏を取る習慣。

2. **「外部要因のせいで直せない」という結論は疑ってかかる**（課題2）: 一度「Wine 本家の既知の問題」と結論づけたが、実際は自分のビルドの構文エラーだった。原因を外部に求める前に、自分の変更が正しく反映されているかを確認する。

3. **解決しない課題も、切り分けができた時点で価値がある**（課題3、課題6）: Space キー問題は未解決だが、「メインスレッドはブロックされていない」「PeekMessage は呼ばれない」「WndProc は再生中に Space を無視する」という複数の誤った思い込みを、それぞれ証拠（トレース・ログ）で訂正できたこと自体が探究のプロセスとして意味を持つ。

4. **1つの症状に複数の原因が重なっているケースを想定する**（課題4）: クラッシュ修正では、1つ直しても別の原因で同じ症状が再現することが繰り返された。都度クラッシュダンプを取得し直し、決めつけずに切り分ける姿勢が必要だった。

5. **機能追加が既存機能を壊すリスクを常に想定する**（課題5）: シェル統合のような広範囲に影響するコード変更では、新機能のテストだけでなく、影響を受けうる既存機能の再確認が必須だった。
